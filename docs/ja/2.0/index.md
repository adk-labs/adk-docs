# ADK 2.0 へようこそ

<div class="language-support-tag">
  <span class="lst-supported">Supported in ADK</span><span class="lst-python">Python v2.0.0</span><span class="lst-go">Go v2.0.0</span>
</div>

ADK 2.0 は、高度な AI エージェントを構築するための強力なツールを導入し、より高い制御性、予測可能性、および信頼性を持って困難なタスクを実行するようにエージェントを構造化するのに役立ちます。ADK 2.0 は Python および Go で利用可能であり、以下の主要機能が含まれています。

-   [**グラフベースのワークフロー**](/ja/graphs/): タスクのルーティングと実行方法をより細かく制御できる、決定論的なエージェント ワークフローを構築します。

-   [**動的ワークフロー**](/ja/graphs/dynamic/): 反復ループや複雑な意思決定ベースの分岐を含む、より複雑なワークフローをコードベースのロジックで構築します。

-   [**協調ワークフロー**](/ja/workflows/collaboration/): コーディネーター エージェントと複数のサブエージェントが連携して動作する、複雑なエージェント アーキテクチャを構築します。

上記のリンク先トピックを確認し、ADK 2.0 による新しいエージェント構築方法をお試しください！

!!! tip "ADK Python v2.0.0 GA リリース"

    ADK Python 2.0 は 2026 年 5 月 19 日付で一般提供（GA）としてリリースされました。

!!! tip "ADK Go v2.0.0 GA リリース"

    ADK Go 2.0 は 2026 年 6 月 30 日付で一般提供（GA）としてリリースされました。

## ADK Python 1.x との互換性

ADK 2.0 は、ADK 1.x リリースで開発されたエージェントと互換性を持つように設計されています。ただし、ADK 1.x プロジェクトを ADK 2.0 にアップグレードする前に注意すべき互換性を破る変更点（breaking changes）がいくつかあります。

!!! warning "破壊的変更: ADK Python 1.x から 2.0 への非互換性"

    ADK Python v2.0.0 で導入された既知の非互換性および破壊的変更がいくつかあります。アップグレードする前にこれらの変更を確認し、必要に応じて対応措置を講じてください。

ADK 2.0 リリースではワークフロー ランタイム（Workflow Runtime）が導入され、ADK は階層型エージェント実行エンジンからグラフベースの実行エンジンへと移行しました。この新しいアーキテクチャでは、エージェント、ツール、および関数はワークフロー グラフ内の個々の *ノード（nodes）* として評価されます。ADK 1.x からアップグレードする場合は、本番アプリケーションの円滑な移行のために以下の破壊的変更および移行手順を確認してください。

### イベント スキーマとカスタム セッション ストレージ

ADK 2.0 では、グラフの状態とワークフロー出力を追跡するために、コア ***Event*** スキマに `node_info` および `output` フィールドが新しく追加されました。

*   **カスタム セッション ストレージ:** 厳格な列構造（rigid columns）を使用して独自の SQL または NoSQL データベースにセッションを保存するなど、独自の `BaseSessionService` を実装している場合は、データベース スキーマを新しいフィールドに対応させるために更新する必要があります。2.0 ***Event*** を 1.x の固定テーブルに挿入しようとすると、挿入エラーまたは ORM 逆シリアル化エラーが発生します。*ただし、カスタム セッション サービスがイベントを明示的な列にマッピングするのではなく、シリアル化された JSON ブロブ（blob）として保存している場合は、スキーマを更新する必要はありません。*
*   **厳格な JSON 検証:** デプロイメントに `additionalProperties: false` の設定を含む厳格な JSON スキーマ検証を行うダウンストリーム API ゲートウェイ、モバイル クライアント、または Web フロントエンドが含まれている場合、期待されるスキーマが更新されるまで 2.0 イベントは拒否されます。

**移行のアクション:** すべての Event ペイロードで `node_info` および `output` フィールドを受信して保存するように、データベース スキーマおよびダウンストリーム クライアント検証ツールを更新してください。2.0 セッションを共有データベースに書き込む前に、すべてのリーダー アプリケーションが 2.0 フォーマットを処理できるように更新されていることを確認してください。

### エージェントの実行: BaseAgent から BaseNode へ

ADK 1.x では、エージェントは独立した実行エンジンでした。ADK 2.0 では、***BaseAgent*** クラスは ***BaseNode*** を継承するようになりました。エージェントは新しいワークフロー グラフ エンジン内で個々の *ノード* として評価されます。

*   **実行ドライバーのカスタム オーバーライド:** ABC コントラクトが変更されました。`_run_async_impl()` や `generate_content()` などの 1.x 抽象メソッドのカスタム オーバーライドは、もはや正しい実行方法ではありません。ワークフロー グラフ エンジンはこれらのレガシー オーバーライドを完全にバイパスします。これらのメソッドをオーバーライドしてカスタム テレメトリや状態管理を注入している場合、それらの呼び出しは無効化されます。

**移行のアクション:** カスタム実行ロジックを `run()` のオーバーライドから外部に移動してください。代わりに、標準化された `BeforeAgentCallback` および `AfterAgentCallback` インターフェースを利用して、実行ライフサイクルにカスタム ロジックを安全に注入してください。

### コンテキストとコールバック: インプレース変異（In-Place Mutation）

フレームワークをバイパスしてイベントを手動で追加することは、もはや安全ではありません。

*   **イベントの直接追加:** ADK 1.x では、一部の開発者が `context.session.events.append(custom_event)` を介してセッションにイベントを直接追加していました。ADK 2.0 では、ワークフロー ランナーが状態、グラフ ルーティング、およびストリーミングを管理するためにイベントの放出を厳格に制御する必要があります。セッション リストに手動で追加すると、グラフ エンジンをバイパスし、決定性が損なわれます。

**移行のアクション:** イベントをセッションに直接追加しないでください。また、`enqueue_event` を直接使用しないでください。フレームワークが永続性、ルーティング、およびストリーミングをネイティブに管理できるように、ノードまたはエージェント内から明示的にイベントを `yield` する必要があります。

### エラー処理と自動再試行

ADK 2.0 フレームワークは、自動再試行、テレメトリ、および HITL（Human-in-the-Loop）の一時停止を有効にするために、例外を自動的にキャッチするようになりました。

*   **`Try...except` と `BaseException`:** ADK 1.x ではフレームワークにネイティブの自動再試行機能がなかったため、開発者はツール内部に手動で `try...except` ループを記述することがよくありました。ADK 2.0 でツールを移行する際に、広範な `except Exception:` ブロックをツール内に残しておくと、フレームワークから障害が隠蔽され、そのステップに対する新しい 2.0 の自動再試行メカニズムが永続的に無効化されます。さらに、`BaseException` をキャッチすると意図せず `NodeInterruptedError` がトラップされ、HITL 入力のためにワークフローを一時停止する機能が損なわれます。

**移行のアクション:** フレームワークが構成された ***RetryConfig***（例: `RetryConfig(max_attempts=3)`）に対して評価できるように、ツールから標準の例外が伝播するのを許可してください。例外を明示的に再発生させる場合を除き、***BaseException*** をキャッチしないでください。

その他の ADK Python 1.0 から 2.0 への非互換性に遭遇した場合は、[イシュー トラッカー](https://github.com/google/adk-python/issues/new?template=bug_report.md&labels=v2) から報告してください。

### ADK Python 1.x のインストール {#install}

ADK を更新したいが、まだ ADK 2.0 に更新する準備ができていない場合は、以下に示すようにインストール時に ADK バージョンを指定するか、互換性のあるリリース `~=` 演算子を使用してください。ADK 1.0 のシステム要件は以下の通りです。

*   **Python 3.10** 以降
*   パッケージ インストール用の `pip`

最新バージョンの ADK 1.x をインストールするには、以下の手順に従ってください。

1.  Python 仮想環境を有効にします。

2.  ADK 1.x 用の互換性のあるリリース `~=` 演算子を使用して pip でパッケージをインストールします。

    ```bash
    pip install "google-adk~=1.0"
    ```

## ADK Go 1.x との互換性

ADK Go 2.0 は、ADK Go 1.x リリースで開発されたエージェントと互換性を持つように設計されています。ただし、ADK Go 1.x プロジェクトを ADK Go 2.0 にアップグレードする前に注意すべき破壊的変更がいくつかあります。

!!! warning "破壊的変更: ADK Go 1.x から 2.0 への非互換性"

    ADK Go v2.0.0 で導入された既知の非互換性および変更点があります。アップグレードする前にこれらの変更を確認してください。

ADK Go 2.0 リリースではワークフロー ランタイムが導入され、ADK Go は階層型エージェント実行エンジンからグラフベースの実行エンジンへと移行しました。

### モジュール インポート パス

ADK Go 2.0 は新しいメジャー バージョンのモジュール パスを使用します。Go ソース ファイルおよび `go.mod` ファイルのすべてのインポート パスを更新する必要があります。

*   **1.x インポート パス:** `google.golang.org/adk`
*   **2.0 インポート パス:** `google.golang.org/adk/v2`

**移行のアクション:** `go get google.golang.org/adk/v2` を実行し、ソース ファイル内のすべてのインポート文を `google.golang.org/adk/...` から `google.golang.org/adk/v2/...` に更新してください。

### エージェントの実行: Agent インターフェースの変更

ADK Go 1.x では、エージェントは `Run` メソッドを提供することで `agent.Agent` インターフェースを実装していました。ADK Go 2.0 では、エージェントは新しいワークフロー グラフ エンジン内の個々の *ノード* として評価されます。

**移行のアクション:** カスタム実行ロジックを標準化された `BeforeAgentCallback` および `AfterAgentCallback` フックに移動し、実行ライフサイクルに安全に注入してください。

### イベント作成: `session.NewEvent` シグネチャの変更

`session.NewEvent` は、最初の引数として `context.Context` を必要とするようになりました。

```go
// 以前 (ADK Go 1.x)
ev := session.NewEvent(ctx.InvocationID())
// または
ev := session.NewEventWithContext(ctx, ctx.InvocationID())

// 以降 (ADK Go 2.0)
ev := session.NewEvent(ctx, ctx.InvocationID())
```

**移行のアクション:** すでにスコープ内にあるコンテキストを `session.NewEvent` の最初の引数として渡してください。

### イベント スキーマとカスタム セッション ストレージ

ADK Go 2.0 では、グラフ ルーティング、ワークフロー状態、および HITL 一時停止をサポートするために、コア ***Event*** 構造体に 5 つの新しいフィールドが追加されました。

| Go フィールド | シリアル化名 | 目的 |
|---|---|---|
| `IsolationScope string` | `isolationScope` | LLM プロンプト履歴でこのイベントを表示できるエージェント コンテキストを制限します。 |
| `Routes []string` | `Routes` | 条件付きエッジ ディスパッチを駆動するためにノードから放出されるルーティング キー。 |
| `RequestedInput *RequestInput` | `RequestedInput` | ワークフロー ノードが人間の入力のために一時停止していることを示します。 |
| `Output any` | `Output` | ワークフロー ノードからの汎用データ出力。 |
| `NodeInfo *NodeInfo` | `nodeInfo` | イベントを放出したノードを識別するワークフロー ノード メタデータ。 |

**移行のアクション:** すべての Event ペイロードで 5 つの新しいフィールドを受信して保存するように、データベース スキーマおよびダウンストリーム クライアント検証ツールを更新してください。

その他の ADK Go 1.0 から 2.0 への非互換性に遭遇した場合は、[イシュー トラッカー](https://github.com/google/adk-go/issues/new?template=bug_report.md&labels=v2) から報告してください。

### ADK Go 1.x のインストール {#install-go}

ADK Go 1.x を引き続き使用したい場合は、1.x リリース ラインに依存関係を固定してください。

```shell
go get google.golang.org/adk@v1
```
