    세션: 개별 대화 추적하기¶

    ADK에서 지원Python v0.1.0TypeScript v0.2.0Go v0.1.0Java v0.1.0

    소개에 이어, Session에 대해 자세히 알아보겠습니다. "대화 스레드"라는 개념을 다시 생각해 보세요. 매번 문자 메시지를 처음부터 새로 시작하지 않는 것처럼, 에이전트도 진행 중인 상호작용에 대한 컨텍스트가 필요합니다. Session은 이러한 개별 대화 스레드를 추적하고 관리하기 위해 특별히 설계된 ADK 객체입니다.

    Session 객체¶

    사용자가 에이전트와 상호작용을 시작하면, SessionService는 Session 객체(google.adk.sessions.Session)를 생성합니다. 이 객체는 하나의 특정 채팅 스레드와 관련된 모든 것을 담는 컨테이너 역할을 합니다. 주요 속성은 다음과 같습니다.

    • 식별 정보 (id, appName, userId): 대화를 위한 고유 레이블입니다.
      • id: 이 특정 대화 스레드에 대한 고유 식별자로, 나중에 대화를 검색하는 데 필수적입니다. SessionService 객체는 여러 개의 Session을 처리할 수 있으며, 이 필드는 우리가 참조하는 특정 세션 객체를 식별합니다. 예: "test_id_modification".
      • app_name: 이 대화가 속한 에이전트 애플리케이션을 식별합니다. 예: "id_modifier_workflow".
      • userId: 대화를 특정 사용자와 연결합니다.
    • 히스토리 (events): 이 특정 스레드 내에서 발생한 모든 상호작용(Event 객체 - 사용자 메시지, 에이전트 응답, 도구 실행)의 시간순 목록입니다.
    • 세션 상태 (state): 진행 중인 이 특정 대화에만 관련된 임시 데이터를 저장하는 공간입니다. 상호작용 중에 에이전트를 위한 임시 메모장 역할을 합니다. state를 사용하고 관리하는 방법은 다음 섹션에서 자세히 다룰 것입니다.
    • 활동 추적 (lastUpdateTime): 이 대화 스레드에서 마지막으로 이벤트가 발생한 시간을 나타내는 타임스탬프입니다.

    예시: 세션 속성 검토¶

     from google.adk.sessions import InMemorySessionService, Session

 # 속성을 검토하기 위해 간단한 세션 생성
 temp_service = InMemorySessionService()
 example_session = await temp_service.create_session(
     app_name="my_app",
     user_id="example_user",
     state={"initial_key": "initial_value"} # 상태는 초기화될 수 있습니다
 )

 print(f"--- 세션 속성 검토 ---")
 print(f"ID (`id`):                {example_session.id}")
 print(f"애플리케이션 이름 (`app_name`): {example_session.app_name}")
 print(f"사용자 ID (`user_id`):         {example_session.user_id}")
 print(f"상태 (`state`):           {example_session.state}") # 참고: 여기서는 초기 상태만 표시됩니다
 print(f"이벤트 (`events`):         {example_session.events}") # 초기에는 비어 있습니다
 print(f"마지막 업데이트 (`last_update_time`): {example_session.last_update_time:.2f}")
 print(f"---------------------------------")

 # 정리 (이 예시에서는 선택 사항)
 temp_service = await temp_service.delete_session(app_name=example_session.app_name,
                             user_id=example_session.user_id, session_id=example_session.id)
 print("temp_service의 최종 상태 - ", temp_service)

}

 System.out.printf("상태 (`state`): %s%n", exampleSession.state());

    SessionService로 세션 관리하기¶

    위에서 본 것처럼, 일반적으로 Session 객체를 직접 생성하거나 관리하지 않습니다. 대신 SessionService를 사용합니다. 이 서비스는 대화 세션의 전체 생명주기를 책임지는 중앙 관리자 역할을 합니다.

    핵심 책임은 다음과 같습니다.

    • 새 대화 시작: 사용자가 상호작용을 시작할 때 새로운 Session 객체를 생성합니다.
    • 기존 대화 재개: 에이전트가 중단된 부분부터 계속할 수 있도록 특정 Session을 ID를 사용해 검색합니다.
    • 진행 상황 저장: 새로운 상호작용(Event 객체)을 세션의 히스토리에 추가합니다. 이는 세션 state가 업데이트되는 메커니즘이기도 합니다('상태' 섹션에서 자세히 설명).
    • 대화 목록 조회: 특정 사용자와 애플리케이션에 대한 활성 세션 스레드를 찾습니다.
    • 정리: 대화가 끝나거나 더 이상 필요하지 않을 때 Session 객체와 관련 데이터를 삭제합니다.

    SessionService 구현체¶

    ADK는 다양한 SessionService 구현체를 제공하므로, 필요에 가장 적합한 스토리지 백엔드를 선택할 수 있습니다.

    1. InMemorySessionService

      • 작동 방식: 모든 세션 데이터를 애플리케이션의 메모리에 직접 저장합니다.
      • 영속성: 없음. 애플리케이션이 다시 시작되면 모든 대화 데이터가 손실됩니다.
      • 필요 사항: 추가 사항 없음.
      • 적합한 경우: 빠른 개발, 로컬 테스트, 예제 및 장기적인 영속성이 필요하지 않은 시나리오.
       from google.adk.sessions import InMemorySessionService
 session_service = InMemorySessionService()

       import "google.golang.org/adk/session"

 inMemoryService := session.InMemoryService()

       import com.google.adk.sessions.InMemorySessionService;
 InMemorySessionService exampleSessionService = new InMemorySessionService();

    2. VertexAiSessionService

      • 작동 방식: 세션 관리를 위해 API 호출을 통해 Google Cloud Vertex AI 인프라를 사용합니다.
      • 영속성: 있음. 데이터는 Vertex AI Agent Engine을 통해 안정적이고 확장 가능하게 관리됩니다.
      • 필요 사항:
        • Google Cloud 프로젝트 (pip install vertexai)
        • 이 단계에 따라 구성할 수 있는 Google Cloud Storage 버킷.
        • 이 튜토리얼에 따라 설정할 수 있는 Reasoning Engine 리소스 이름/ID.
        • Google Cloud 프로젝트가 없고 VertexAiSessionService를 무료로 사용해보고 싶다면, 세션 및 메모리 무료로 사용해보기를 참조하세요.
      • 적합한 경우: Google Cloud에 배포된 확장 가능한 프로덕션 애플리케이션, 특히 다른 Vertex AI 기능과 통합할 때.
      # 필요: pip install google-adk[vertexai]
# 추가로 GCP 설정 및 인증 필요

session_service = VertexAiSessionService(project=PROJECT_ID, location=LOCATION)
# 서비스 메서드 호출 시 REASONING_ENGINE_APP_NAME 사용, 예:
# session_service = await session_service.create_session(app_name=REASONING_ENGINE_APP_NAME, ...)

    3. DatabaseSessionService

      ADK에서 지원Python v0.1.0Go v0.1.0

      • 작동 방식: 관계형 데이터베이스(예: PostgreSQL, MySQL, SQLite)에 연결하여 세션 데이터를 테이블에 영구적으로 저장합니다.
      • 영속성: 있음. 데이터는 애플리케이션 재시작 후에도 유지됩니다.
      • 필요 사항: 구성된 데이터베이스.
      • 적합한 경우: 직접 관리하는 안정적이고 영속적인 스토리지가 필요한 애플리케이션.
      from google.adk.sessions import DatabaseSessionService
# 로컬 SQLite 파일을 사용하는 예시:
db_url = "sqlite:///./my_agent_data.db"
session_service = DatabaseSessionService(db_url=db_url)

    올바른 SessionService를 선택하는 것은 에이전트의 대화 기록과 임시 데이터가 어떻게 저장되고 유지되는지를 정의하는 핵심입니다.

    세션 생명주기¶

    대화의 한 턴(turn) 동안 Session과 SessionService가 함께 작동하는 단순화된 흐름은 다음과 같습니다.

    1. 시작 또는 재개: 애플리케이션은 SessionService를 사용하여 create_session을 호출(새 채팅)하거나 기존 세션 ID를 사용해야 합니다.
    2. 컨텍스트 제공: Runner는 적절한 서비스 메서드에서 Session 객체를 가져와, 에이전트가 해당 세션의 state 및 events에 접근할 수 있도록 합니다.
    3. 에이전트 처리: 사용자가 에이전트에게 쿼리를 보냅니다. 에이전트는 쿼리와 잠재적으로 세션의 state 및 events 히스토리를 분석하여 응답을 결정합니다.
    4. 응답 및 상태 업데이트: 에이전트는 응답을 생성하고 (잠재적으로 state에서 업데이트할 데이터를 플래그 지정) Runner는 이를 Event로 패키징합니다.
    5. 상호작용 저장: Runner는 session과 새로운 event를 인자로 하여 sessionService.append_event(session, event)를 호출합니다. 서비스는 Event를 히스토리에 추가하고, 이벤트 내의 정보를 기반으로 스토리지의 세션 state를 업데이트합니다. 세션의 last_update_time도 업데이트됩니다.
    6. 다음 준비: 에이전트의 응답이 사용자에게 전달됩니다. 업데이트된 Session은 이제 SessionService에 의해 저장되어 다음 턴을 위해 준비됩니다(이 주기는 보통 현재 세션에서 대화를 계속하며 1단계부터 다시 시작됩니다).
    7. 대화 종료: 대화가 끝나면, 더 이상 필요하지 않은 경우 애플리케이션은 sessionService.delete_session(...)을 호출하여 저장된 세션 데이터를 정리합니다.

    이러한 주기는 SessionService가 각 Session 객체와 관련된 히스토리와 상태를 관리함으로써 대화의 연속성을 어떻게 보장하는지 보여줍니다.
